# 项目指南技能

该技能用于快速接手陌生项目：把仓库整理成实用的项目学习指南。**开始分析前，必须先让用户选择分析模式。**

跨平台：在支持 skill 机制的平台上可直接加载；在其他平台上可将 `PROMPT.md` 的内容作为系统提示词复制使用。

## 第一步：让用户选择分析模式

向用户展示三个选项（每次激活必须执行）：

1. **📄 浅层分析** — 快速 Markdown 报告（`project-guide/project-guide.md`），含技术栈、结构树、核心流程、埋点清单、阅读路线。适合快速了解概况。
2. **🌐 深层分析** ⭐ 推荐 — 两阶段：Phase 1 秒出骨架卡片 → Phase 2 按需生成交互式 HTML 指南（`project-guide/` 目录，约 18 个文件）。适合深入理解和团队 onboarding。
3. **🔍 聚焦分析** — 指定模块/功能逐文件逐行深度剖析（`project-guide/{模块名}-focus.md` 或 `.html`），含影响范围评估。适合接手特定功能或重构前评估。

用户未选择时默认深层分析。选聚焦分析时确认目标模块和输出格式（md/html）。

**智能推荐：** 展示选项前，先快速检测项目规模（`rg --files | grep -v -E '(node_modules|\.git|dist|build|\.next|vendor|__pycache__)' | wc -l`）：
- 🟢 小型（<150）→ 推荐深层分析
- 🔴 大型（400+）→ 推荐先浅层了解全局，再对关键模块用聚焦分析
- ⚫ 超大型（1000+）→ 推荐先浅层，再逐个对子项目用聚焦分析

## 大型项目优化

文件数 ≥ 150 时，按 🟢<150 / 🟡150-400 / 🔴400-1000 / ⚫1000+ 四级自适应调整：文件采样率、代码提取数量、目录树深度。核心原则：**先快照、再聚焦、渐进深入**。

> 完整策略见 `references/large-projects.md`。

## 增量分析（跨会话缓存）

如果目标项目之前已被分析过（存在 `project-guide/js/guide-data.js`），支持增量更新——只分析变更文件，大幅降低重复分析成本。

### 缓存检测

分析开始前（规模检测之后、模式选择之前），检查目标项目中是否存在 `project-guide/js/guide-data.js`。如果存在：

1. 读取文件中的 `_meta` 字段（`analyzedAt`、`gitCommit`、`fileCount`、`analysisMode`、`projectRoot`）
2. 确认 `projectRoot` 与当前项目路径匹配
3. 获取当前 HEAD commit hash：`git rev-parse HEAD`
4. 计算变更：`git diff --stat <上次commit>..HEAD`

### 向用户展示

```markdown
📌 检测到 {N} 天前的分析缓存

| 项目 | 值 |
| --- | --- |
| 上次分析 | {日期} · commit `{短hash}` · ~{fileCount} 文件 |
| 当前 HEAD | commit `{短hash}` |
| 变更文件 | +{新增} / ~{修改} / -{删除} |

是否增量更新？（只分析变更文件，预计节省 {估算}% 时间）

🔄 **Y. 增量更新** — 保留不变部分的已有分析，只重新分析变更文件
🔃 **N. 全量重新分析** — 忽略缓存，从头分析
```

### 增量分析策略

用户选 Y 后执行：

**保留不变的部分（直接复用已有数据）：**
- 项目概览（名称、描述、成熟度）
- 技术栈（除非 manifest/lockfile 变更）
- 架构概览（除非入口文件或核心模块结构变更）
- 开发约定
- 新人阅读路线
- 快速导航（追加新模块，移除已删除模块）

**重新分析变更的文件：**
- 对每个变更文件：读取 → 更新/新增/移除代码分析卡片
- 更新结构树：新增/修改/删除对应节点及其优先级标注
- 重新搜索变更文件中的埋点调用 → 合并到埋点清单
- 重新评估依赖方向（如果 import 语句有变更）
- 重新生成概念地图（如果路由/服务/状态目录有变更）

**更新元数据：**
- `_meta.analyzedAt` → 当前时间
- `_meta.gitCommit` → 当前 HEAD
- `_meta.fileCount` → 当前文件数

**重新生成受影响的文件：**
- `js/guide-data.js`（完整数据，增量合并后）
- 仅覆盖有变更的 `js/sections/*.js` 文件
- CSS、core.js、index.html 等框架文件无需重写

### 降级条件

以下情况自动降级为全量重新分析（不询问用户）：
- 未找到 `_meta` 字段或 `gitCommit` 为空（旧版格式不支持增量）
- `git diff --stat` 返回 100+ 变更文件（变更量过大，增量收益低）
- 项目根路径不匹配（不同项目的缓存）
- manifest/lockfile 变更（依赖变更可能影响全局）
- 目录结构顶层发生变更（如 `src/` → `packages/` 重构）

### 浅层/聚焦模式的增量

- **浅层模式**：如果存在 `project-guide/project-guide.md`，读取其中的生成日期和 commit 信息，同样支持增量
- **聚焦模式**：不支持增量（每次聚焦的目标模块不同，缓存无复用价值）

## 模式一：浅层分析

对全项目做摘要级分析，输出一个 Markdown 文件。

**分析内容：** README、清单文件、顶层目录、入口文件、核心模块、配置、测试。项目结构树 + 依赖方向图、核心流程追踪、开发约定、埋点清单、快速导航索引。

**输出：** `project-guide/project-guide.md`（模板见 `references/output-templates.md`）。

## 模式二：深层分析（两阶段）

### Phase 1：即时扫描（10-30 秒）

并行执行 7 个快速任务：项目规模、README 身份识别、技术栈清单、顶层结构、导出全景、依赖概览、埋点 SDK 检测。输出**项目骨架卡片**——项目类型、技术栈、入口点、顶层结构、发现的业务领域、风险标志。

骨架卡片末尾附带 Phase 2 菜单：

- **A. 全部深入** — 完整深层分析，生成交互式 HTML 指南
- **B. 展开模块** — 对选定领域执行聚焦分析（可多次选择）
- **C. 只看维度** — 只看特定维度（埋点/数据流/架构/代码分析）
- **D. 到此为止** — 骨架已够用，结束分析

60 秒未响应默认选 A。

> 骨架卡片输出模板见 `references/output-templates.md`。

### Phase 2A：全部深入

浅层分析的全部内容 + 关键业务代码提取与逐行分析 → 生成模块化 `project-guide/` 目录。

**代码提取要求：** 提取入口初始化、路由定义、API 服务层、状态管理结构、埋点调用、核心业务流程（小型 5-7 段 → 大型 2-3 段）。每段代码附带文件路径、行号、功能说明、逐行分析、输入输出、注意事项。不提取密钥/Token/密码/连接串。

**HTML 指南架构：** Section 注册表模式（`ProjectGuide.registerSection()`），数据与表现分离（`guide-data.js` 单一数据源），兼容 `file://` 协议。13 个 section + 核心框架文件，约 18 个文件。

> 完整模板、CSS、section 代码见 `references/html-guide-template.md`。

**功能要求：** 左侧导航 + 搜索 + 交互式目录树 + Mermaid 图（CDN，无网络降级） + 代码展示面板（文件路径 + 代码 + 逐行分析 + 一键复制） + 暗/亮主题 + 响应式 + 键盘快捷键（Ctrl+K 搜索、Ctrl+/ 侧边栏、Esc 关闭）。

### Phase 2B：展开模块

对选定领域执行聚焦分析（等同于模式三），生成 `project-guide/{模块名}-focus.md` 或 `.html`。用户可多次选择逐个展开。

### Phase 2C：只看维度

只生成指定维度的深度内容（如只看埋点/数据流/架构）。输出量比完整深层分析少 60-80%。

### Phase 2D：到此为止

Phase 1 骨架已足够。后续可重新触发。

## 模式三：聚焦分析

确认目标模块（功能名/目录路径/关键词）和输出格式（默认 md）。

**六步流程：**
1. **圈定边界** — 搜索所有相关文件，列出完整清单，区分模块内/外部依赖
2. **模块内部结构** — 文件清单、内部引用关系、对外 import、外部对模块的 import、依赖岛 ASCII 图
3. **逐文件深度分析** — 完整读取（不抽样），提取关键代码 + 逐行分析，标注内部调用关系，识别潜在问题
4. **模块级数据流** — 追踪进入/离开模块的完整处理链路，绘制 Mermaid 时序图
5. **模块级埋点** — 搜索模块内所有埋点，分析完整性，标注缺失
6. **影响范围评估** — 列出所有依赖本模块的外部文件，按高/中/低分级，给出安全重构切入点

**输出章节：** 模块定位 → 文件清单 → 内部结构 → 外部依赖 → 逐文件代码分析 → 数据流追踪 → 埋点清单 → 风险和技术债 → 影响范围评估。

> 输出模板见 `references/output-templates.md`。

## 操作原则

- 先检查再下结论。区分：已观察事实、文档声明、运行验证、推断、待确认问题。
- **开始分析前必须让用户选择模式。**
- 关键结论绑定证据：文件路径、配置、命令输出、测试结果。
- 不输出密钥、令牌、连接串，只描述用途和位置。
- 按职责、流程、边界、约定和风险解释，不逐文件流水账。
- 未执行的命令要说明原因。
- 面向中文项目默认输出中文报告。

## 核心流程

1. **展示三选一**（必须）。聚焦分析需确认目标模块和输出格式。
2. **检测项目规模**（必须）：`rg --files | grep -v -E '...' | wc -l`，按四级判定，后续步骤按等级调整深度。
3. **检测分析缓存**：检查目标项目中是否存在 `project-guide/js/guide-data.js` 或 `project-guide/project-guide.md`。如存在且 git 可用，执行增量分析检测（见「增量分析」章节），询问用户是否增量更新。
4. **（深层专属）Phase 1 即时扫描**：并行 7 个快速任务，10-30 秒输出骨架卡片 + Phase 2 菜单。
5. **（深层专属）Phase 2 菜单**：等待用户选择，60 秒默认 A。选 D 则结束。
6. 仓库盘点（只读）：顶层文件/目录、文档、manifest、lockfile、配置、脚本、测试、CI、容器、环境模板。跳过生成物/vendor/缓存/构建目录。
7. 识别技术栈：语言、框架、包管理器、运行时、构建工具。
8. 判断项目形态：前端/后端/全栈/库/CLI/monorepo/移动端/混合。按形态加载 `references/project-types.md` 对应章节。
9. 找入口点：启动命令、应用根、路由/菜单、API client、服务入口、状态根。
10. 追踪主路径：输入 → 路由/分发 → 组件/服务 → 外部依赖 → 状态/输出。
11. 绘制项目结构树（规范见 `references/project-structure.md`）：ASCII 目录树 + P0-P3 优先级 + ★●▲⚠ 标记 + 依赖方向图 + 循环依赖识别。大型项目自动限制深度。
12. 构建概念地图（深层/聚焦）：从路由/服务/状态/目录名提取业务概念 → 标注关键文件和关联 → 生成 Mermaid LR 概念关系图。小型 3-5 个，大型 8-15 个。
13. 分析代码文件（规范见 `references/project-structure.md`）：P0/P1 文件生成摘要卡片。大型项目按规模等级采样。
14. 识别埋点（规范见 `references/tracking-analytics.md`）：统一 grep 搜索 → 整理清单 → 覆盖缺口。
15. 识别开发约定 + 构建快速导航索引。
16. 按模式输出产物。如果为增量更新，仅覆盖有变更的 section 文件。

## 证据标签

- **已观察** — 通过读取文件确认
- **文档声明** — 文档中写明但未独立验证
- **运行验证** — 通过命令/测试/API 确认
- **推断** — 由结构/命名/调用关系得出
- **待确认** — 需要用户或环境确认

## 完成标准

- 项目规模已检测，分析深度已按等级调整
- 项目定位、技术栈、入口点已明确
- 项目结构树已生成（含优先级和依赖方向）
- （深层/聚焦）概念地图已构建（业务概念 + Mermaid 图 + 关键文件）
- 关键代码文件已分析（大型项目已按等级采样）
- 埋点事件已搜索整理
- 快速导航索引已构建
- **浅层：** `project-guide/project-guide.md` 已写入
- **深层 Phase 1：** 骨架卡片已输出 + Phase 2 菜单已展示
- **深层 Phase 2A：** 关键业务代码已提取分析 + `project-guide/` 目录全部生成
- **深层 Phase 2B/C/D：** 按对应路径完成
- **聚焦：** 模块边界已圈定 + 逐文件分析完成 + 影响范围已评估 + 产物已写入
- **大型项目：** 产出中已标注采样率和可深入方向
- **增量更新：** `_meta` 已更新（analyzedAt、gitCommit、fileCount）；仅变更文件被重新分析；不变部分的已有数据被保留；受影响的 section 文件已重写
- HTML 中无密钥/Token/密码等敏感信息
