# 项目指南技能

该技能用于快速接手陌生项目：把仓库整理成实用的项目学习指南。**开始分析前，展示合并决策面板（规模 + 缓存 + 模式推荐），用户做一次选择即可。**

跨平台：在支持 skill 机制的平台上可直接加载；在其他平台上可将 `PROMPT.md` 的内容作为系统提示词复制使用。

## 第一步：项目规模检测 + 缓存检查 + 智能推荐（合并为一次交互）

**分析开始前，并行执行 3 项检测（10 秒内完成）：**

1. **项目规模**：`rg --files | grep -v -E '(node_modules|\.git|dist|build|\.next|vendor|__pycache__)' | wc -l`
2. **分析缓存**：检查 `project-guide/js/guide-data.js`（深层）或 `project-guide/project-guide.md`（浅层）是否存在
3. **Git 可用性**：`git rev-parse HEAD 2>/dev/null`

然后将所有信息合并为**一次性决策面板**展示给用户：

```markdown
📊 项目规模：{🟢小型/🟡中型/🔴大型/⚫超大} · ~{N} 文件
{如果有缓存：📌 {N} 天前的 {模式} 分析缓存 · +{A}/~{B}/-{C} 变更}

建议：{基于规模和缓存的智能推荐}

请选择分析模式：
1. 📄 浅层分析 — Markdown 报告，适合快速了解概况
2. 🌐 深层分析 {⭐ 推荐} — 交互式 HTML 指南，适合团队 onboarding
3. 🔍 聚焦分析 — 指定模块逐文件逐行剖析，适合接手特定功能
   （选此项请说明目标模块，如「聚焦分析 src/modules/auth」）
```

**推荐逻辑：**
- 🟢 小型（<150）→ 深层分析 ⭐
- 🟡 中型（150-400）→ 深层分析 ⭐ 或先浅层再聚焦
- 🔴 大型（400-1000）→ 浅层分析 ⭐ 或聚焦指定模块
- ⚫ 超大型（1000+）→ 聚焦分析 ⭐（先浅层了解全局，再逐个模块聚焦）

**增量分析自动判定（不询问用户的情况）：**
- 变更文件 < 20 → 自动增量更新（展示提示但不等确认）
- 变更文件 > 100 或 lockfile/manifest 变更或目录顶层重构 → 自动全量重新分析
- 变更文件 20-100 → 展示缓存信息，询问是否增量（仅此情况需用户决策）

用户 30 秒未响应时，按推荐项自动执行。

## 大型项目优化

文件数 ≥ 150 时，按 🟢<150 / 🟡150-400 / 🔴400-1000 / ⚫1000+ 四级自适应调整：文件采样率、代码提取数量、目录树深度。核心原则：**先快照、再聚焦、渐进深入**。

> 完整策略见 `references/large-projects.md`。

## 增量分析（跨会话缓存）

已有分析缓存时，优先增量更新而非全量重分析。

### 自动判定规则

| 条件 | 行为 |
| --- | --- |
| 变更 < 20 文件，且 lockfile/manifest 未变 | **自动增量**（展示提示，不等待确认） |
| 变更 20-100 文件 | **询问用户** 是否增量 |
| 变更 > 100 文件 | **自动全量**（不询问，增量收益低） |
| lockfile/manifest/顶层目录结构变更 | **自动全量**（不询问，影响全局） |
| 无 `_meta` 字段或 `gitCommit` 为空 | **自动全量**（旧版格式不支持增量） |
| 无 Git 仓库 | **自动全量**（跳过增量逻辑，不询问） |

### 增量分析执行

**保留不变的部分：** 项目概览、技术栈（lockfile 未变时）、架构概览、开发约定、阅读路线、快速导航

**重新分析变更的文件：** 读取变更文件 → 更新代码分析卡片 → 更新结构树节点 → 重搜埋点（变更文件范围内） → 更新概念地图（如果路由/服务/状态目录有变更）

**更新产物：** `guide-data.js`（合并）、仅覆盖有变更的 `sections/*.js` 文件、更新 `_meta`（analyzedAt、gitCommit、fileCount）。框架文件（CSS、core.js、index.html）无需重写。

**聚焦模式：** 不支持增量（每次目标模块不同）

## 模式一：浅层分析

对全项目做摘要级分析，输出一个 Markdown 文件。

**分析内容：** README、清单文件、顶层目录、入口文件、核心模块、配置、测试。项目结构树 + 依赖方向图、核心流程追踪、开发约定、埋点清单、快速导航索引。

**输出：** `project-guide/project-guide.md`（模板见 `references/output-templates.md`）。

---

## 模式二：深层分析（两阶段，Phase 2 自动启动）

### Phase 1：即时扫描（10-30 秒）→ 展示骨架卡片

并行执行 7 个快速任务：项目规模、README 身份识别、技术栈清单、顶层结构、导出全景、依赖概览、埋点 SDK 检测。输出**项目骨架卡片**——项目类型、技术栈、入口点、顶层结构、发现的业务领域、风险标志。

骨架卡片末尾附带提示（非阻塞，不等用户确认）：

> 🔄 正在生成完整 HTML 项目指南...（预计 {N} 分钟）
> 💡 如需聚焦特定模块，随时说「聚焦 {模块名}」即可切换为聚焦分析模式。

### Phase 2：全面深入分析（Phase 1 完成后自动启动）

**无需等待用户确认，Phase 1 结束后直接进入 Phase 2。** 用户可在 Phase 2 进行中随时中断并切换为聚焦模式。

浅层分析的全部内容 + 关键业务代码提取与逐行分析 → 生成模块化 `project-guide/` 目录。

**代码提取要求：** 提取入口初始化、路由定义、API 服务层、状态管理结构、埋点调用、核心业务流程（小型 5-7 段 → 大型 2-3 段）。每段代码附带文件路径、行号、功能说明、逐行分析、输入输出、注意事项。不提取密钥/Token/密码/连接串。

**HTML 指南架构：** Section 注册表模式（`ProjectGuide.registerSection()`），数据与表现分离（`guide-data.js` 单一数据源），兼容 `file://` 协议。13 个 section + 核心框架文件，约 18 个文件。

> 完整模板、CSS、section 代码见 `references/html-guide-template.md`。

**功能要求：** 左侧导航 + 搜索 + 交互式目录树 + Mermaid 图（CDN，无网络降级） + 代码展示面板（文件路径 + 代码 + 逐行分析 + 一键复制） + 暗/亮主题 + 响应式 + 键盘快捷键（Ctrl+K 搜索、Ctrl+/ 侧边栏、Esc 关闭）。

**用户可在生成过程中随时切换模式：**
- 说「聚焦 {模块名}」→ 中断深层分析，切换到聚焦模式分析指定模块
- 说「只看 {维度}」→ 只生成指定维度的深度内容（埋点/数据流/架构/代码分析）
- 说「停下」→ 保留已生成的部分内容

## 模式三：聚焦分析

确认目标模块（功能名/目录路径/关键词）和输出格式（默认 md）。

**六步流程：**
1. **圈定边界** — 搜索所有相关文件，列出完整清单，区分模块内/外部依赖
2. **模块内部结构** — 文件清单、内部引用关系、对外 import、外部对模块的 import、依赖岛 ASCII 图
3. **逐文件深度分析** — 完整读取（不抽样），提取关键代码 + 逐行分析，标注内部调用关系，识别潜在问题
4. **模块级数据流** — 追踪进入/离开模块的完整处理链路，绘制 Mermaid 时序图
5. **模块级埋点** — 搜索模块内所有埋点，分析完整性，标注缺失
6. **影响范围评估** — 列出所有依赖本模块的外部文件，按高/中/低分级，给出安全重构切入点

**输出章节：** 模块定位 → 文件清单 → 内部结构 → 外部依赖 → 逐文件代码分析 → 数据流追踪 → 埋点清单 → 风险和技术债 → 影响范围评估。

> 输出模板见 `references/output-templates.md`。

## 操作原则

- 先检查再下结论。区分：已观察事实、文档声明、运行验证、推断、待确认问题。
- **开始分析前展示合并决策面板（规模+缓存+推荐），一次交互完成所有决策。30 秒未响应按推荐项自动执行。**
- 关键结论绑定证据：文件路径、配置、命令输出、测试结果。
- 不输出密钥、令牌、连接串，只描述用途和位置。
- 按职责、流程、边界、约定和风险解释，不逐文件流水账。
- 未执行的命令要说明原因。
- 面向中文项目默认输出中文报告。

## 核心流程

1. **并行检测（10 秒内）**：规模检测 + 缓存检查 + Git 可用性，一次性完成。
2. **展示合并决策面板**：规模 + 缓存状态 + 智能推荐 + 三种模式选项。用户只做一次选择。增量更新按自动判定规则执行（变更 < 20 自动增量、> 100 自动全量、中间才问）。30 秒未响应按推荐项自动执行。
3. **（深层专属）Phase 1 即时扫描**：并行 7 个快速任务 → 骨架卡片 + 自动启动 Phase 2 提示。
4. **（深层专属）Phase 2 自动启动**：不等用户确认，直接进入全面分析。用户可随时中断并切换模式。
5. 仓库盘点（只读）：顶层文件/目录、文档、manifest、lockfile、配置、脚本、测试、CI、容器、环境模板。跳过生成物/vendor/缓存/构建目录。
6. 识别技术栈：语言、框架、包管理器、运行时、构建工具。
7. 判断项目形态：前端/后端/全栈/库/CLI/monorepo/移动端/混合。按形态加载 `references/project-types.md` 对应章节。
8. 找入口点：启动命令、应用根、路由/菜单、API client、服务入口、状态根。
9. 追踪主路径：输入 → 路由/分发 → 组件/服务 → 外部依赖 → 状态/输出。
10. 绘制项目结构树（规范见 `references/project-structure.md`）：ASCII 目录树 + P0-P3 优先级 + ★●▲⚠ 标记 + 依赖方向图 + 循环依赖识别。大型项目自动限制深度。
11. 构建概念地图（深层/聚焦）：从路由/服务/状态/目录名提取业务概念 → 标注关键文件和关联 → 生成 Mermaid LR 概念关系图。小型 3-5 个，大型 8-15 个。
12. 分析代码文件（规范见 `references/project-structure.md`）：P0/P1 文件生成摘要卡片。大型项目按规模等级采样。
13. 识别埋点（规范见 `references/tracking-analytics.md`）：统一 grep 搜索 → 整理清单 → 覆盖缺口。
14. 识别开发约定 + 构建快速导航索引。
15. 按模式输出产物。如果为增量更新，仅覆盖有变更的 section 文件。

### 快速失败条件

以下情况**主动建议降级**，不硬撑全量分析：

| 信号 | 建议 |
| --- | --- |
| 2000+ 文件 | 先浅层概览，再逐个模块聚焦 |
| 多语言混合无明确主语言 | 分语言/子项目分别分析 |
| 无入口点、无 manifest、无文档 | 仅做结构梳理，标注"需人工确认" |
| 单文件超过 5000 行 | 跳过逐行分析，仅做段落摘要 |
| 用户已连续 2 次选择聚焦模式 | 不再推荐深层全量分析 |

## 证据标签

- **已观察** — 通过读取文件确认
- **文档声明** — 文档中写明但未独立验证
- **运行验证** — 通过命令/测试/API 确认
- **推断** — 由结构/命名/调用关系得出
- **待确认** — 需要用户或环境确认

## 完成标准

- 项目规模已检测，分析深度已按等级调整
- 项目定位、技术栈、入口点已明确
- 项目结构树已生成（含优先级和依赖方向）
- （深层/聚焦）概念地图已构建（业务概念 + Mermaid 图 + 关键文件）
- 关键代码文件已分析（大型项目已按等级采样）
- 埋点事件已搜索整理
- 快速导航索引已构建
- **浅层：** `project-guide/project-guide.md` 已写入
- **深层：** Phase 1 骨架卡片已输出 + Phase 2 自动启动并完成全面分析 → `project-guide/` 目录全部生成
- **聚焦：** 模块边界已圈定 + 逐文件分析完成 + 影响范围已评估 + 产物已写入
- **大型项目：** 产出中已标注采样率和可深入方向
- **增量更新：** `_meta` 已更新（analyzedAt、gitCommit、fileCount）；仅变更文件被重新分析；不变部分的已有数据被保留；受影响的 section 文件已重写
- HTML 中无密钥/Token/密码等敏感信息
- 未触发快速失败条件；如触发已主动建议降级方案
