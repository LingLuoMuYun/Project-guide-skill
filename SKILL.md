# 项目指南技能

该技能用于快速接手陌生项目：把仓库整理成实用的项目学习指南。**在开始分析前，先让用户选择分析模式**。

该技能适用于任何 AI Agent（Claude Code、Codex、Cursor、ChatGPT、Gemini 等）。在支持 skill 机制的平台上可以直接作为 skill 加载；在其他平台上可以将 `PROMPT.md` 的内容作为系统提示词复制使用。

## 首先：让用户选择分析模式

**每次激活此技能时，第一步必须向用户展示三个选项：**

```markdown
请选择分析模式：

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

📄 选项 1：浅层分析（快速生成 Markdown 报告）
   输出：`project-guide/project-guide.md`
   内容：技术栈 · 目录结构 · 入口点 · 核心流程
         模块职责 · 埋点清单 · 开发约定 · 风险 · 阅读路线
   适合：快速了解项目概况 · 交接概要 · 初次接触

🌐 选项 2：深层分析（两阶段：即时骨架 → 按需深入）⭐ 推荐
   输出：Phase 1 项目骨架卡片（秒出）+ Phase 2 按需生成 `project-guide/` 目录
   内容：先秒出技术栈/入口/结构/业务领域概览，再按你选择的方向深入
         + Mermaid 架构图 + 交互式目录树 + 可搜索埋点表格
         + 快速导航 + 暗/亮主题切换 + 模块化可扩展架构
   适合：深入理解项目 · 团队 onboarding · 长期维护参考

🔍 选项 3：聚焦分析（针对特定模块/功能深入分析）
   输出：`project-guide/{模块名}.md` 或 `project-guide/{模块名}.html`
   内容：针对指定模块/功能/目录的深度剖析
         · 该模块的完整文件清单和依赖图
         · 逐文件代码展示与逐行分析
         · 该模块的数据流/调用链追踪
         · 该模块的埋点事件
         · 该模块的风险点和技术债
         · 修改该模块的影响范围评估
   适合：需要深入理解某个具体模块 · 接手特定功能 · 局部重构前评估

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

请输入 1 / 2 / 3，或直接说"浅层"/"深层"/"聚焦分析"。
如选聚焦分析，请同时说明目标模块，例如：
  · "聚焦分析 用户认证模块"
  · "针对 src/services/order/ 做深入分析"
  · "只看支付流程，输出 html"
```

用户选择后，按对应模式执行。用户没有选择时，默认执行深层分析（选项 2）。

**智能推荐（可选）：** 展示模式选项前，先快速扫描项目规模（`rg --files | wc -l`），根据结果在提示中附加推荐建议：
- 🟢 小型项目 → "建议选 2（深层分析），Phase 1 秒出骨架后可直接选 A 全部深入"
- 🔴 大型项目 → "建议先选 1（浅层分析）快速了解全局，再对关键模块使用 3（聚焦分析）"
- ⚫ 超大型/Monorepo → "建议先选 1 了解整体结构，再逐个对子项目使用 3（聚焦分析）"

## 大型项目优化策略

分析 300+ 文件的大型项目时，按**先快照、再聚焦、渐进深入**原则自适应调整分析深度。

核心机制：分析开始前执行项目规模检测 → 按 🟢<150 / 🟡150-400 / 🔴400-1000 / ⚫1000+ 四级自动调整文件采样率、代码提取数量、目录树深度。涵盖 9 个通用加速技巧（并行读取、统一 grep、批量导出提取、一次导入图等）。

> 完整策略、分级参数和命令模板见 `references/large-projects.md`。

## 模式一：浅层分析 → 生成 `project-guide/project-guide.md`

### 分析范围

- README、清单文件、顶层目录、入口文件、核心模块、配置、测试
- 项目结构树 + 依赖方向图
- 核心流程追踪
- 开发约定识别
- 埋点事件梳理（简要表格）
- 快速导航索引

### 输出

在目标项目根目录生成 **一个 Markdown 文件**：`project-guide/project-guide.md`。

### 章节结构

详见 `references/output-templates.md` 中的「浅层分析模板」。

### 完成标准

- 项目概览、技术栈、结构树已输出
- 核心流程和模块职责已梳理
- 埋点事件已列表
- 快速导航索引已构建
- 文件已写入目标项目根目录

## 模式二：深层分析 → 两阶段分析（即时骨架 + 按需深入）

深层分析采用**两阶段分析**模式，解决"长时间等待却不知道 AI 在干什么"的痛点：

- **Phase 1（即时扫描）**：10-30 秒内并行扫描，输出**项目骨架卡片**——类型、技术栈、入口点、顶层结构、发现的业务领域、风险标志
- **Phase 2（按需深入）**：用户看到骨架后，选择深入方向——全部深入 / 展开特定模块 / 只看特定维度 / 到此为止

### 第一阶段：即时扫描（Phase 1）

**目标**：10-30 秒内给出项目骨架，让用户快速确认"这是什么项目、用什么技术栈、入口在哪"。

**并行扫描任务（全部同时执行，墙钟时间 = 最慢任务 ≈ 10-15 秒）：**

```bash
# 任务 1：项目规模（1-2秒）
rg --files | grep -v -E '(node_modules|\.git|dist|build|\.next|vendor|__pycache__)' | wc -l

# 任务 2：项目身份（1秒）
cat README.md 2>/dev/null | head -60

# 任务 3：技术栈与命令（1秒）
cat package.json 2>/dev/null  # 或 pyproject.toml / go.mod / Cargo.toml / pom.xml

# 任务 4：顶层结构（2秒）
ls -la | grep -v -E '(node_modules|\.git|dist)'

# 任务 5：导出全景（3-5秒，快速了解核心入口和模块）
rg "^export (async )?(function|class|const|interface|type|default)" --no-filename | head -80

# 任务 6：依赖概览（3-5秒，了解模块间引用关系）
rg "^import.*from" --no-filename | head -100

# 任务 7：埋点 SDK 检测（2-3秒）
rg -n -i "gtag|mixpanel|sensors|sentry|aegis|analytics|_hmt|growingio|track\(|trackEvent|logEvent|reportEvent|pageView|sendEvent|useTrack|data-track" | head -30
```

**Phase 1 输出格式（项目骨架卡片）：**

```markdown
## 🏷️ 项目骨架 — {项目名称}

| 属性 | 值 |
| --- | --- |
| 名称 | {从 package.json/README 提取} |
| 类型 | {前端/后端/全栈/CLI/...} |
| 规模 | {🟢🟡🔴⚫} · ~{文件数} 文件 |
| 入口 | `{入口文件}` → `{路由/主模块}` |
| 启动 | `{dev命令}` → {端口/URL} |
| 构建 | `{build命令}` → `{输出目录}` |

### 🛠️ 技术栈
{技术标签云，每个标签标注证据文件}

### 📁 顶层结构
{顶层目录 + 说明（不超过 10 行）}

### 🧩 发现的业务领域
（从路由/服务/状态/目录名推断）
1. {领域A} — {关键文件}
2. {领域B} — {关键文件}
...

### ⚠ 风险标志
- {未找到环境变量模板 / 有循环依赖 / 无测试 / ...}

---
🐍 以上扫描耗时约 {N} 秒。接下来想深入了解什么？

🔍 **A. 全部深入** — 完整深层分析，生成交互式 HTML 指南（需 {估算耗时}）
🧩 **B. 展开模块** — 选择感兴趣的领域深入分析（输入编号，如 `B 1,3`）
📊 **C. 只看维度** — 只看特定维度：埋点/数据流/架构/代码分析
✅ **D. 到此为止** — 骨架已提供足够信息，不需继续
```

### 第二阶段：按需深入（Phase 2）

用户选择后按对应路径执行。如 60 秒内未响应则默认选 A（确保自动化脚本不阻塞）。

**选 A：全部深入** — 执行完整深层分析，生成模块化 `project-guide/` 目录（约 18 个文件）。以下为 Phase 2A 的完整流程：

> 完整模块化架构文档见 `references/html-guide-template.md`

### Phase 2A 分析范围

浅层分析的全部内容 + 以下增强：

1. **关键业务代码提取**：从 P0/P1 文件中提取核心代码片段
2. **代码逐行分析**：对提取的代码添加注释和分析说明
3. **模块化交互式 HTML**：生成结构化的项目指南目录（约 18 个文件）

### 代码提取与分析要求（核心差异）

深层分析必须提取并分析以下类型的代码：

**必须提取的代码片段：**
- 入口文件的核心初始化逻辑
- 路由定义（完整路由表）
- API 服务层的关键接口调用
- 状态管理（Store/Context 的核心结构）
- 埋点调用的实际代码
- 核心业务流程的关键函数/组件
- 复杂/易出错的逻辑段

**每段代码必须附带：**
- 文件路径和行号
- 代码功能说明（这段代码做什么）
- 逐行/逐段分析注释（关键行的解释）
- 输入输出说明
- 注意事项（边界条件、已知问题、性能考量）

**代码提取安全原则：**
- 不提取密钥、Token、密码、连接串
- 不提取完整的用户敏感数据处理逻辑
- 可以提取模块声明、函数签名、业务逻辑结构

### HTML 指南目录结构

深层分析产出为 `project-guide/` 目录，约 18 个文件：


### HTML 核心架构

- **Section 注册表模式**：每个章节通过 `ProjectGuide.registerSection({ id, title, icon, priority, render })` 注册。添加新分析维度只需新增 section 文件 + 数据字段 + 一行 `<script>` 标签。
- **数据与表现分离**：`guide-data.js` 集中管理 `window.__PROJECT_DATA__`，section 文件只负责渲染。
- **兼容 `file://` 协议**：所有 JS 通过 `<script src>` 加载，无需本地服务器。
- 完整模板见 `references/html-guide-template.md`。

### HTML 功能要求

- 左侧固定导航栏自动从已注册 section 构建 + 顶部搜索栏
- 交互式目录树（展开/折叠/全部展开/全部折叠）
- Mermaid 架构图和时序图（CDN 加载 mermaid@10，无网络时降级为静态文本）
- **代码展示面板**：代码块顶部显示文件路径，每段代码后紧跟分析注释块
- 代码块一键复制
- 全文搜索（模块/文件/函数/埋点/代码内容）
- 暗色/亮色主题切换（偏好存 localStorage，首次跟随系统）
- 响应式布局（桌面/平板/手机）
- 键盘快捷键（Ctrl+K 搜索、Ctrl+/ 侧边栏、Esc 关闭）
- Section 隔离：一个章节渲染失败不影响其他章节

**选 B：展开模块** — 对选定领域执行聚焦分析：

## 模式三：聚焦分析 → 针对指定模块/功能深入分析

当用户选择聚焦分析时，先确认两个信息：

**1. 目标模块/功能**（用户可能以以下方式指定）：
- 功能名称："用户认证模块""支付流程""权限系统"
- 目录路径："src/services/order/""apps/admin/pages/"
- 文件范围："所有与 shopping-cart 相关的文件"
- 关键词："涉及 websocket 的代码"

**2. 输出格式**（用户未指定时默认 md）：
- md → 生成 `project-guide/{模块名}-focus.md`
- html → 生成 `project-guide/{模块名}-focus.html`

### 分析范围

聚焦分析的核心原则：**范围缩小、深度加深**。在窄范围内做到比深层分析更细致。

**第一步：圈定分析边界**
- 根据用户指定的模块/路径/关键词，搜索所有相关文件
- 列出完整的文件清单（绝对不遗漏）
- 确定边界：哪些文件属于本模块，哪些是外部依赖

**第二步：绘制模块内部结构**
- 该模块的文件清单和目录结构
- 模块内部文件的相互引用关系
- 模块对外的 import（依赖了谁）
- 外部对模块的 import（谁依赖了它）
- 用 ASCII 图绘制该模块的依赖岛

**第三步：逐文件深度分析**
对模块内的每个文件执行：
- 完整读取文件内容（聚焦模式下不抽样）
- 生成代码摘要：职责、核心逻辑
- **提取关键代码片段并逐行分析**（比深层模式更细致）
- 标注与本模块其他文件的调用关系
- 识别潜在问题（技术债、性能、安全）

**第四步：模块级数据流追踪**
- 追踪进入该模块的数据/请求的完整处理链路
- 追踪该模块对外发起的调用和输出
- 绘制模块级 Mermaid 时序图

**第五步：模块级埋点分析**
- 搜索该模块内所有埋点调用
- 分析埋点设计的完整性
- 标注可能缺失埋点的关键路径

**第六步：影响范围评估**
- 列出所有依赖本模块的外部文件
- 评估修改该模块的风险和波及范围
- 给出安全的重构切入点建议

### 输出产物

**md 格式：**
```text
project-guide/{模块名}-focus.md
```

包含章节：
1. 模块定位（这个模块解决什么问题）
2. 文件清单（完整列表，含每个文件的职责）
3. 模块内部结构（目录树 + 引用关系图）
4. 外部依赖关系（谁依赖它 / 它依赖谁）
5. 逐文件代码分析（每个文件的代码片段 + 逐行分析）
6. 数据流追踪（该模块的核心处理链路）
7. 模块埋点清单
8. 风险和技术债
9. 修改影响范围评估

**html 格式：**
```text
project-guide/{模块名}-focus.html
```

在 md 内容基础上，使用交互式 HTML 模板渲染，包含：
- 模块级导航侧边栏
- 代码展示面板（代码 + 分析注释）
- Mermaid 依赖图和时序图
- 搜索（模块内文件/函数/变量）
- 暗/亮主题切换

### 聚焦模式示例

用户说："聚焦分析 用户认证模块，输出 html"

执行：
1. 搜索项目中的 auth/login/认证/权限相关文件
2. 圈定：`src/pages/Login/`, `src/services/authService.ts`, `src/store/userStore.ts`, `src/hooks/useAuth.ts`, `src/components/AuthGuard.tsx`
3. 逐个深度分析每个文件
4. 追踪：用户输入凭据 → 前端校验 → API 调用 → token 存储 → 状态更新 → 路由守卫检查
5. 生成 `project-guide/用户认证模块-focus.html`

## 操作原则

- 先检查再下结论。区分已观察事实、文档声明、运行验证、合理推断和待确认问题。
- **开始分析前必须让用户选择浅层/深层/聚焦模式**。
- 优先依据仓库真实结构，不套用泛化架构标签。
- 关键结论要绑定证据：文件路径、配置、命令输出、测试结果或文档。
- 不输出密钥、令牌、连接串等敏感值，只描述其用途和位置。
- 避免逐文件流水账。按职责、流程、边界、约定和风险解释。
- 未执行的命令要说明未执行及原因。
- 用户未指定语言时，面向中文项目默认输出简洁中文报告。

## 核心流程

1. **向用户提供浅层/深层/聚焦三选一**（必须）。如用户选聚焦分析，还需确认目标模块和输出格式（md/html）。
2. **检测项目规模**（必须）：
   ```bash
   rg --files | grep -v -E '(node_modules|\.git|dist|build|\.next|vendor|__pycache__)' | wc -l
   ```
   根据文件数判定规模等级（🟢<150 / 🟡150-400 / 🔴400-1000 / ⚫1000+），后续所有步骤按「大型项目优化策略」中的对应等级调整分析深度。
3. **（深层模式专属）Phase 1 即时扫描**：选择选项 2 后，立即并行执行 7 个快速扫描任务（规模、身份、技术栈、顶层结构、导出全景、依赖概览、埋点 SDK 检测），10-30 秒内输出**项目骨架卡片**给用户。
4. **展示 Phase 2 菜单**：骨架卡片末尾附带 A/B/C/D 选项，等待用户选择深入方向。60 秒未响应默认选 A。选 D 则分析结束。
5. 使用快速只读方式盘点仓库（浅层/聚焦模式从步骤 3 开始；深层 Phase 2A 继续以下步骤）：
   - 顶层文件和目录
   - 文档、manifest、lockfile、配置、脚本、测试、CI、容器、环境模板
   - 需要跳过的生成物、vendor、缓存、构建目录
6. 识别技术栈：语言、框架、包管理器、运行时、构建工具、基础设施文件。
7. 判断项目形态：前端、后端、全栈、库、CLI、monorepo、数据/机器学习、基础设施、移动/桌面、插件或混合系统。
8. 找入口点：启动命令、应用根、路由/菜单定义、API client、服务入口、状态根、测试入口。
9. 追踪主路径：输入/用户动作 -> 路由/分发 -> 组件/服务 -> 外部依赖 -> 状态/输出。
10. **绘制项目结构树**：带注释的 ASCII 目录树、P0-P3 优先级、★●▲⚠ 标记、依赖方向图、循环依赖识别。大型项目自动限制深度。
9. **构建概念地图**（深层/聚焦模式）：从路由、服务、状态、目录名中提取业务概念，标注每个概念的关键文件、优先级、关联概念，生成 Mermaid 概念关系图（LR 方向）。概念数量：小型 3-5 个，中型 5-10 个，大型 8-15 个。
10. **分析代码文件内容**：对 P0/P1 文件生成摘要卡片。大型项目按规模等级采样，不逐文件全读。
11. **识别埋点与追踪**：搜索 SDK 引用、调用模式、整理清单、覆盖缺口。
12. 识别开发约定：目录布局、命名、状态使用、样式、权限、测试。
13. 评估验证情况。
14. **构建快速导航索引**。
15. **按所选模式输出**：
    - 浅层 → Markdown 报告 `project-guide/project-guide.md`
    - 深层 → Phase 1 已输出骨架卡片 → 按用户 Phase 2 选择执行：A=提取关键业务代码 → 逐段分析 → 写入 `project-guide/` 目录 / B=执行聚焦分析 / C=输出指定维度 / D=结束
    - 聚焦 → **圈定模块边界** → 逐文件深度分析 → 写入 `project-guide/{模块名}-focus.md` 或 `.html`

## 证据标签

- **已观察**：通过读取文件确认。
- **文档声明**：文档中写明，但未独立验证。
- **运行验证**：通过命令、测试或 API 输出确认。
- **推断**：由结构、命名或调用关系得出的合理结论。
- **待确认**：需要用户、后端、运维或运行环境确认。

## 完成标准

收尾前应回答或标记待确认：

- 项目规模是否已检测？分析深度是否按规模等级调整？
- （深层模式）Phase 1 并行扫描是否已执行？骨架卡片是否秒出？
- 这是什么项目，提供什么价值？
- 如何安装、启动、构建、测试、部署或打包？
- 主要入口和运行边界在哪里？
- 最重要的数据流和控制流是什么？
- 项目结构树是否已生成（含优先级标注和依赖方向）？（大项目：深度是否已限制？）
- 概念地图是否已构建？（业务概念是否从路由/服务/状态中提取？Mermaid 关系图是否生成？概念详情卡片是否包含关键文件？）
- 关键代码文件是否已生成摘要？（大项目：是否使用了批量导出提取代替逐个读取？）
- 埋点事件是否已搜索并整理？（大项目：是否使用了统一 grep？）
- 快速导航索引是否已构建？
- **浅层模式**：`project-guide/project-guide.md` 是否已写入目标项目根目录？
- **深层模式 Phase 1**：项目骨架卡片是否已输出？（包含技术栈/入口/结构/业务领域/风险标志）Phase 2 菜单（A/B/C/D）是否已展示？
- **深层模式 Phase 2A**（用户选全部深入）：关键业务代码是否已提取并逐段分析？`project-guide/` 目录（约 18 个文件）是否已全部生成？
- **深层模式 Phase 2B**（用户选展开模块）：聚焦分析是否已执行？模块报告是否已写入？（同聚焦模式标准）
- **深层模式 Phase 2C**（用户选只看维度）：指定维度是否已深入分析？产物是否已输出？
- **深层模式 Phase 2D**（用户选到此为止）：Phase 1 骨架是否已提供足够信息快速了解项目？
- **聚焦模式**：模块边界是否已圈定？是否逐文件完成深度分析？影响范围是否已评估？（大项目：是否使用了一次导入图代替逐文件反向搜索？）产物是否已写入 `project-guide/{模块名}-focus.md` 或 `.html`？
- **大项目专项**：是否在产出中标注了"快速模式"限制（如采样率、跳过的文件数、可后续深入的方向）？

## 输出质量自检

分析完成后，Agent 必须自检以下项目：

1. **可用性**：生成的 HTML 能否在浏览器中正常打开？Mermaid 图是否能渲染？
2. **完整性**：是否遗漏了 P0 入口文件或核心模块？
3. **安全性**：`guide-data.js` 中是否包含密钥、Token、密码？
4. **规模合规**：输出深度是否与检测到的项目规模匹配？大项目是否标注了采样率和可深入方向？
5. **一致性**：文件路径、行号引用是否与实际代码一致？
