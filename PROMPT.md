## 触发场景

当用户提出以下意图时立即激活：
- **快速上手：** 快速入手、快速上手、快速了解、认识项目、看懂项目、入门
- **指引指南：** 项目指引、项目指南、入门指南、生成项目指南
- **解析分析：** 项目解析、代码解析、架构解析、项目分析、代码分析
- **梳理整理：** 梳理项目、结构梳理、目录树、结构树、项目概览
- **接手交接：** 接手项目、项目交接、交接文档
- **埋点追踪：** 埋点梳理、埋点分析、追踪分析
- **聚焦分析：** 聚焦分析、针对xx模块、深入分析xx、只看xx模块
- **英文：** project guide、onboarding、focus on、deep dive into

## ⚠️ 第一步：让用户选择分析模式（必须）

每次激活必须首先展示三个选项，等待用户选择：

1. **📄 浅层分析** — 快速 Markdown 报告（`project-guide/project-guide.md`）。技术栈 · 结构树 · 核心流程 · 模块职责 · 埋点清单 · 阅读路线。适合快速了解概况。
2. **🌐 深层分析** ⭐ 推荐 — 两阶段：Phase 1 秒出项目骨架卡片 → Phase 2 按需生成交互式 HTML 指南（`project-guide/` 目录，约 18 个文件）。含 Mermaid 架构图 + 交互式目录树 + 代码展示与逐行分析 + 搜索 + 暗/亮主题。适合深入理解和团队 onboarding。
3. **🔍 聚焦分析** — 指定模块/功能逐文件逐行深度剖析（`project-guide/{模块名}-focus.md` 或 `.html`），含影响范围评估。适合接手特定功能或重构前评估。

用户未选择时默认深层分析。选聚焦分析时需确认目标模块和输出格式。

**智能推荐：** 展示前先检测项目规模：`rg --files | grep -v -E '(node_modules|\.git|dist|build|\.next|vendor|__pycache__)' | wc -l`。小型(<150)推荐深层，大型(400+)推荐先浅层再聚焦，超大型(1000+)推荐浅层后逐个子项目聚焦。

## 大型项目优化

文件数 ≥ 150 时按四级自适应调整：

| 规模 | 文件数 | 策略 |
| --- | --- | --- |
| 🟢 小型 | < 150 | 全量分析 |
| 🟡 中型 | 150-400 | 智能采样，P1 每目录 ≤5 文件，目录树 ≤4 层 |
| 🔴 大型 | 400-1000 | 大幅采样 + P1 每目录 ≤3 文件且仅读前 100 行 + 目录树 ≤3 层 |
| ⚫ 超大型 | 1000+ | 快速概览 → 建议用聚焦分析逐模块深入 |

核心加速技巧：统一埋点搜索（一条 grep 覆盖全部模式）、批量导出提取（`rg "^export"` 代替逐个读文件）、一次导入图（`rg "^import.*from"` 构建完整依赖图）、并行维度分析、模板直接复制。

## 增量分析（跨会话缓存）

如果目标项目之前已被分析过（存在 `project-guide/js/guide-data.js`），支持增量更新——只分析变更文件，大幅降低重复分析成本。

### 缓存检测

分析开始前，检查目标项目中是否存在 `project-guide/js/guide-data.js`。如果存在：

1. 读取 `_meta` 字段：`analyzedAt`、`gitCommit`、`fileCount`、`analysisMode`、`projectRoot`
2. 确认 `projectRoot` 与当前项目路径匹配
3. 获取当前 HEAD：`git rev-parse HEAD`
4. 计算变更：`git diff --stat <上次commit>..HEAD`

### 向用户展示

向用户展示缓存信息（分析时间、commit 对比、变更文件统计），让用户选择：

- **Y. 增量更新** — 保留不变部分的已有分析，只重新分析变更文件（预计节省大量时间）
- **N. 全量重新分析** — 忽略缓存，从头分析

### 增量分析策略（用户选 Y 后执行）

**保留不变的部分（直接复用）：**
- 项目概览（名称、描述、成熟度）
- 技术栈（除非 manifest/lockfile 变更）
- 架构概览（除非入口文件或核心模块结构变更）
- 开发约定、新人阅读路线
- 快速导航（追加新模块，移除已删除模块）

**重新分析变更的文件：**
- 对每个变更文件：读取 → 更新/新增/移除代码分析卡片
- 更新结构树：新增/修改/删除对应节点
- 重新搜索变更文件中的埋点调用 → 合并到埋点清单
- 重新评估依赖方向（如果 import 语句有变更）
- 重新生成概念地图（如果路由/服务/状态目录有变更）

**更新元数据：** `_meta.analyzedAt` → 当前时间；`_meta.gitCommit` → 当前 HEAD；`_meta.fileCount` → 当前文件数。

**重新生成：** 更新 `js/guide-data.js`（完整合并后的数据），仅覆盖有变更的 `js/sections/*.js` 文件。CSS、core.js、index.html 等框架文件无需重写。

### 降级条件（自动全量重新分析，不询问）

- 未找到 `_meta` 或 `gitCommit` 为空（旧版格式）
- `git diff --stat` 返回 100+ 变更文件
- 项目根路径不匹配
- manifest/lockfile 变更
- 目录结构顶层变更（如 `src/` → `packages/` 重构）

### 浅层/聚焦模式的增量

- **浅层模式**：如果存在 `project-guide/project-guide.md` 且含日期和 commit 信息，同样支持增量
- **聚焦模式**：不支持增量（每次聚焦的目标模块不同，缓存无复用价值）

## 操作原则

- 先检查再下结论。区分：已观察事实、文档声明、运行验证、推断、待确认。
- **必须先让用户选择模式。**
- 关键结论绑定证据：文件路径、配置、命令输出、测试结果。
- 不输出密钥、令牌、连接串，只描述用途和位置。
- 按职责、流程、边界、约定和风险解释，不逐文件流水账。
- 面向中文项目默认输出中文报告。

## 通用分析流程

### 0. 项目规模检测（必须首先执行）

```bash
rg --files | grep -v -E '(node_modules|\.git|dist|build|\.next|vendor|__pycache__)' | wc -l
```

按 🟢<150 / 🟡150-400 / 🔴400-1000 / ⚫1000+ 判定，后续步骤按等级调整深度。

### 0b. 检测分析缓存

检查目标项目中是否存在 `project-guide/js/guide-data.js` 或 `project-guide/project-guide.md`。如存在且 git 可用，执行增量分析检测（见「增量分析」章节），询问用户是否增量更新。

### 0c. Phase 1 即时扫描（深层模式专属）

选择深层分析后，先并行执行 7 个快速任务，10-30 秒内输出项目骨架卡片：

```bash
# 并行执行：
# 1. 项目规模（上一步已获得）
# 2. cat README.md | head -60                          → 项目身份
# 3. cat package.json / pyproject.toml / go.mod 等     → 技术栈与命令
# 4. ls -la（排除 node_modules/.git/dist）             → 顶层结构
# 5. rg "^export (async )?(function|class|const|interface|type|default)" --no-filename | head -80  → 导出全景
# 6. rg "^import.*from" --no-filename | head -100      → 依赖概览
# 7. rg -n -i "gtag|mixpanel|sensors|sentry|track\(|trackEvent|logEvent|reportEvent|pageView|sendEvent|useTrack|data-track" | head -30  → 埋点SDK检测
```

骨架卡片格式：

```
## 🏷️ 项目骨架 — {项目名称}

| 属性 | 值 |
| --- | --- |
| 名称 | {从 package.json/README 提取} |
| 类型 | {前端/后端/全栈/CLI/...} |
| 规模 | {🟢🟡🔴⚫} · ~{文件数} 文件 |
| 入口 | `{入口文件}` → `{路由/主模块}` |
| 启动 | `{dev命令}` → {端口/URL} |
| 构建 | `{build命令}` → `{输出目录}` |

### 🛠️ 技术栈
{技术标签云，标注证据文件。按语言/框架/构建/数据库分组}

### 📁 顶层结构
{顶层目录 + 说明，不超过 10 行，标注 P0/P1/P2}

### 🧩 发现的业务领域
（从路由/服务/状态/目录名推断，列出概念名 + 关键文件）

### ⚠ 风险标志
{仅列出阻碍项目启动的严重问题：无 .env.example、循环依赖、无测试等}

---
🕐 以上扫描耗时约 {N} 秒。接下来想深入了解什么？

🔍 **A. 全部深入** — 完整深层分析，生成交互式 HTML 指南（预计 {N} 分钟）
🧩 **B. 展开模块** — 选择感兴趣的领域深入分析（输入编号，如 `B 1,3`）
📊 **C. 只看维度** — 只看特定维度：埋点/数据流/架构/代码分析
✅ **D. 到此为止** — 骨架已够用，不继续
```

用户选择后按对应路径执行。60 秒未响应默认选 A。

### 1. 仓库盘点（只读）

- 顶层文件和目录，跳过 node_modules/.git/dist/build/vendor/__pycache__/.next
- 文档、manifest、lockfile、配置、脚本、测试、CI、容器、环境模板
- 识别技术栈：语言、框架、包管理器、运行时、构建工具
- 判断项目形态：前端/后端/全栈/库/CLI/monorepo/移动端/混合
  - 前端 → 重点关注路由、组件、状态、API client、构建配置
  - 后端 → 重点关注路由、中间件、数据库、缓存、消息队列
  - monorepo → 先盘点所有子项目，再逐个分析

### 2. 入口点

启动命令、应用根、路由定义、API client、服务入口、状态根、测试入口。

### 3. 主路径追踪

输入/用户动作 → 路由/分发 → 组件/服务 → 外部依赖 → 状态/输出。

### 4. 项目结构树

用 ASCII 树形格式生成带注释的完整目录树：
- 按职责分组，标注优先级：P0 必读 / P1 重要 / P2 参考 / P3 可跳过
- 特殊标记：★ 入口、● 核心模块、▲ 配置、⚠ 风险
- 绘制依赖方向图（自上而下的分层依赖）
- 标记循环依赖、上帝模块、孤岛模块
- 大型项目自动限制深度（中型 ≤4 层、大型 ≤3 层）

### 4b. 构建概念地图（深层/聚焦）

- 从路由、服务、状态管理、目录名中提取业务概念
- 每个概念标注：名称、描述、优先级、关键文件路径、关联概念
- 生成 Mermaid graph LR 概念关系图（节点=业务概念，边=依赖/数据流）
- 概念数量：小型 3-5 个 / 中型 5-10 个 / 大型 8-15 个

### 5. 代码文件内容分析

对 P0/P1 文件生成摘要卡片：文件职责、核心导出、内部依赖、被依赖方、关键逻辑、潜在问题。
- 中型项目：P1 每目录 ≤5 个文件，P2 仅列文件名
- 大型项目：P1 每目录 ≤3 个文件且仅读前 100 行，优先用批量导出提取代替逐个读取

### 6. 埋点与追踪识别

一条统一 grep 完成：
```bash
rg -n -i "gtag|mixpanel|sensors|sentry|aegis|analytics|_hmt|growingio|track\(|trackEvent|logEvent|reportEvent|pageView|sendEvent|useTrack|v-track|data-track"
```

整理清单：事件名 / 类别（页面浏览/用户行为/业务流程/错误事件）/ 触发时机 / 所在文件:行号 / 参数 / 状态。识别覆盖缺口。

### 7. 开发约定 + 快速导航索引

目录布局、命名规范、状态使用、样式、权限、测试等约定。
构建三维索引：按功能模块 / 按文件类型 / 常用命令。

12. 快速导航（按模块/按文件类型/常用命令）
13. 新人阅读路线

## 选项 2：深层分析 → 两阶段

### Phase 2A：全部深入 — 生成交互式 HTML 指南

浅层分析全部内容 + 关键业务代码提取与逐行分析 → 模块化 `project-guide/` 目录。

**代码提取要求：**
- 提取：入口初始化、路由定义、API 服务层（2-5 个核心接口）、状态管理结构、埋点调用、核心业务逻辑（1-3 个关键流程）
- 数量：🟢 5-7 段 / 🟡 3-5 段 / 🔴 2-3 段
- 每段附带：文件路径+行号、功能说明、逐行分析、输入输出、调用方、注意事项
- 安全：不提取密钥/Token/密码/连接串

**HTML 输出结构：** `{目标项目根目录}/project-guide/`，约 18 个文件，Section 注册表模式。

```
project-guide/
├── index.html                  # 入口（薄壳，~50 行）
├── css/guide.css               # 所有样式
├── js/
│   ├── guide-core.js           # 核心框架：Section 注册表 + UI 工具
│   ├── guide-data.js           # 项目分析数据（单一数据源 window.__PROJECT_DATA__）
│   └── sections/               # 13 个 Section 渲染器（可独立增删）
│       ├── overview.js         # 1. 项目概览
│       ├── quickstart.js       # 2. 快速开始
│       ├── structure.js        # 3. 项目结构树
│       ├── concept-map.js      # 4. 概念地图 🧭
│       ├── architecture.js     # 5. 架构概览
│       ├── code-analysis.js    # 6. 代码展示与分析 ⭐
│       ├── modules.js          # 7. 核心模块详解
│       ├── dataflow.js         # 8. 数据流追踪
│       ├── tracking.js         # 9. 埋点事件清单
│       ├── workflow.js         # 10. 开发工作流
│       ├── roadmap.js          # 11. 新人阅读路线
│       ├── risks.js            # 12. 风险和注意事项
│       └── appendix.js         # 13. 附录
└── extensions/README.md        # 扩展开发指南
```

数据流：`guide-data.js → window.__PROJECT_DATA__ → 各 section.render(container, data) → DOM`

**功能要求：**
- 左侧固定导航（自动从已注册 section 构建）+ 右侧内容 + 顶部搜索
- 交互式目录树（展开/折叠/全部展开/全部折叠）
- Mermaid 架构图/时序图（CDN mermaid@10，无网络降级为静态文本）
- 代码展示面板：文件路径栏 + 代码块 + 分析注释区 + 一键复制
- 全文搜索（模块/文件/函数/埋点/代码内容）
- 暗色/亮色主题（偏好存 localStorage，首次跟随系统）
- 响应式布局（桌面/平板/手机）
- 键盘快捷键：Ctrl+K 搜索 / Ctrl+/ 侧边栏 / Esc 关闭
- Section 错误隔离：单个 section 渲染失败不影响其他
- 兼容 `file://` 协议：所有 JS 通过 `<script src>` 加载

**代码展示面板结构：**
每段代码渲染为 `.code-panel` 容器：
1. `.code-header` — 文件图标 + 路径:行号 + 复制按钮
2. `pre.code-content` — 实际代码（保留原始缩进）
3. `.code-analysis` — 分析区：功能说明 + 逐行说明列表 + 输入输出 + 调用方 + ⚠ 注意事项

**生成步骤：**
1. `mkdir -p project-guide/css project-guide/js/sections project-guide/extensions`
2. 复制固定模板：`guide.css`、`guide-core.js`、`index.html`、`extensions/README.md`
3. 填充 `guide-data.js`：将所有分析结果写入 `window.__PROJECT_DATA__`
4. 按需生成 `js/sections/*.js`：有数据的 section 保留，无数据的删除对应 `<script>`
5. 告知用户用浏览器打开 `project-guide/index.html`

### Phase 2B：展开模块

对选定领域执行聚焦分析（同选项 3）：圈定相关文件 → 逐文件深度分析 → 生成 `project-guide/{模块名}-focus.md` 或 `.html`。可多次选择。

### Phase 2C：只看维度

只生成指定维度：埋点/数据流/架构/代码分析。输出量比完整深层少 60-80%，速度快 3-5 倍。

### Phase 2D：到此为止

Phase 1 骨架已够用。后续可重新触发。

## 选项 3：聚焦分析 → 指定模块深入

先确认目标（功能名/目录路径/关键词）和输出格式（默认 md）。

### 六步流程

**第一步：圈定边界** — 搜索所有相关文件，列出完整清单，区分模块内/外部依赖。

**第二步：模块内部结构** — 文件清单 + 内部引用关系 + 对外 import + 外部对模块的 import + ASCII 依赖岛图。

**第三步：逐文件深度分析**（核心）— 完整读取每个文件（不抽样），对每个文件输出：
- 文件职责（一句话）
- 核心导出（函数/类/组件列表）
- 关键代码片段 + 逐行分析（比深层模式更细致）
- 内部依赖 + 被依赖方
- 风险点

**第四步：模块级数据流** — 追踪进入/离开模块的完整链路，绘制 Mermaid 时序图。

**第五步：模块级埋点** — 搜索模块内所有埋点，分析完整性，标注缺失。
