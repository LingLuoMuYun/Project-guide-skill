# 大型项目优化策略

分析 300+ 文件的大型项目时，必须启用以下优化。**核心原则：先快照、再聚焦、渐进深入。**

## 第一步：项目规模检测（在所有分析开始前执行）

```bash
# 统计项目规模（排除无关目录）
rg --files | grep -v -E '(node_modules|\.git|dist|build|\.next|vendor|__pycache__)' | wc -l
```

| 规模 | 文件数 | 策略 |
| --- | --- | --- |
| 🟢 小型 | < 150 | 全量分析，不做优化 |
| 🟡 中型 | 150 - 400 | 智能采样，P1 每目录最多 5 个文件 |
| 🔴 大型 | 400 - 1000 | 大幅采样 + 并行读取 + 限制代码提取 |
| ⚫ 超大型 | 1000+ | 快速概览 → 建议用户用聚焦分析逐模块深入 |

## 第二步：按规模自适应调整

### 🟡 中型项目（150-400 文件）

**文件读取优化：**
- P0 文件：全读（不变）
- P1 文件：每目录最多读取 **5 个代表性文件**，按修改频率和文件大小选择
- P2 文件：只列出文件名和一行职责描述，**不读内容**
- P3 文件：仅计数，不逐文件说明

**深层分析调整：**
- 代码提取限制：**3-5 段**（小型项目 5-7 段）
- 目录树深度：**≤4 层**展开，更深层折叠
- Mermaid 图：仅架构图（跳过时序图，除非用户明确需要）

**并行执行：**
- 结构树生成、埋点搜索、入口点分析可**并行启动**
- 多个同优先级文件的读取可**批量并行**（一次读取 3-5 个）

### 🔴 大型项目（400-1000 文件）

在中型策略基础上进一步收紧：

**文件读取优化：**
- P0 文件：全读
- P1 文件：每目录最多 **3 个代表性文件**，仅读前 100 行获取关键导出
- P2 文件：仅列出目录名，不逐文件列举
- P3 文件：跳过

**深层分析调整：**
- 代码提取限制：**2-3 段**，仅提取入口初始化和最核心业务流程
- 目录树：**≤3 层**展开
- Mermaid 图：仅架构图（无时序图）
- 代码分析：从"逐行分析"降级为"段落摘要"

**增量生成（推荐）：**
1. 首轮（快速）：生成 `index.html` + 概览 + 结构树 + 核心模块（首批 4-6 个 section）
2. 告知用户核心内容已生成，询问是否需要补充代码分析、数据流、埋点
3. 按用户指令增量追加

### ⚫ 超大型项目（1000+ 文件）

不建议一开始就做全项目深层分析。应：
1. 执行浅层分析生成快速 Markdown 概览
2. 识别项目中的核心子项目/子模块
3. **建议用户使用模式三（聚焦分析）** 逐模块深入
4. 如果用户坚持深层分析，使用快速深层模式（仅 2 段代码 + 目录树 2 层 + 无 Mermaid）

## 通用加速技巧（所有规模适用）

### 读取优化
1. **并行读取**：多个独立文件同时读取。例如入口文件、配置文件、package.json 可在同一轮并行读取
2. **grep 先行**：先用 grep 搜索关键模式，再决定读哪些文件，避免盲目全读
3. **头部分析**：大文件（>300 行）先读前 50-100 行获取导出和结构，再决定是否读全文
4. **批量导出提取**：对 P1 文件不逐个完整读取，用一条 grep 批量提取所有导出签名：
   ```bash
   rg "^export (async )?(function|class|const|interface|type|default)" src/ --no-filename -n
   ```
   从 ~150 次文件读取降低到 ~30 次（仅 P0 全读 + 少数选中的 P1）

### 搜索优化
5. **统一埋点搜索**：用一条 grep 覆盖所有埋点模式，避免多次扫描同一仓库：
   ```bash
   rg -n -i "gtag|mixpanel|sensors|sentry|aegis|analytics|_hmt|growingio|track\(|trackEvent|logEvent|reportEvent|pageView|sendEvent|useTrack|v-track|data-track"
   ```
   一次扫描获得全部埋点事件，在内存中分类，节省 5-6 次全仓库扫描
6. **一次导入图**（聚焦分析时）：用一条命令构建完整导入图，而非逐文件反向搜索：
   ```bash
   rg -n "^import.*from" --no-filename
   ```
   解析此输出即可获得所有依赖关系，避免对每个文件执行单独的反向依赖搜索

### 生成优化
7. **模板复用**：HTML 生成时，`guide-core.js`、`guide.css`、`extensions/README.md` 是固定模板，直接复制不修改，节省决策时间
8. **section 按需生成**：没有数据的 section 直接跳过，不生成占位文件
9. **并行维度分析**：结构树生成、埋点搜索、入口点分析互不依赖，可并行执行。收集完所有维度结果后再汇总输出
