# 分析检查清单

分析仓库时使用本清单。除非用户明确要求，不要把清单原样输出给用户。

## 仓库表面信息

- 项目名称和文档声明的用途
- 使用的语言和包管理器
- 顶层目录
- 文档、示例、测试、脚本、CI、部署文件
- 环境模板或 `.env.*`
- 应跳过的生成物、vendor、缓存、构建目录
- 单项目、嵌套项目、monorepo 或多语言混合系统

## 入口点

- 本地启动命令
- 主应用、服务端或 CLI 入口文件
- 前端应用根和路由
- 后端路由和 controller
- API client 和 service 层
- 全局状态入口
- 测试、构建、lint 命令
- 后台 worker、队列、定时任务

## 架构

- 主要分层和运行边界
- 输入处理和分发路径
- 核心领域逻辑
- 外部系统适配层
- 错误表示和传播方式
- 公开/稳定契约与内部细节

## 数据流和控制流

- 最重要的用户或系统流程
- 每一步如何转换数据
- 向后传递的 ID、路径、handle 或状态引用
- 外部副作用
- 结果如何返回调用方或 UI

## 状态和契约

- 必需配置
- 读取的环境变量
- 请求/响应 schema 或类型
- 内存中的运行时状态
- 持久化状态、缓存、文件、数据库、localStorage/sessionStorage
- 可能过期的数据

## 验证

- 现有测试和测试框架
- 当前环境是否能运行测试
- 手动冒烟测试
- 实际执行过的命令
- 通过检查能证明什么
- 仍未验证什么

## 风险扫描

- 配置含义不清
- 硬编码端口或路径
- 缺少错误处理
- 隐藏外部状态
- 校验薄弱
- 文档过期
- 测试缺失
- 大型未类型化 payload
- 竞态条件或重复副作用
- 平台特定假设
- 构建/部署脚本有副作用
- 本地命令使用接近生产的默认配置

## 项目结构树

- 顶层目录按职责分组（核心代码、配置、测试、脚本、文档、构建产物）
- 关键文件标记（入口文件 ★、核心模块 ●、配置文件 ▲、风险文件 ⚠）
- 每个目录/文件的优先级标注（P0 必读 / P1 重要 / P2 参考 / P3 可跳过）
- 依赖关系方向（自上而下的分层依赖是否清晰）
- 循环依赖或"上帝模块"识别
- 可忽略的目录（node_modules, dist, build, .git, vendor, __pycache__ 等）

### 目录树标注模板

```text
项目根
├── src/                        # ★ P0 核心源代码
│   ├── main.ts                 # ★ P0 应用入口
│   ├── App.vue                 # ★ P0 根组件
│   ├── router/                 # ● P0 路由定义
│   ├── pages/                  # ● P1 页面组件
│   ├── components/             # ● P1 通用组件
│   ├── services/               # ● P1 API 服务层
│   ├── store/                  # ● P1 状态管理
│   ├── hooks/                  # ● P1 自定义 Hook
│   ├── utils/                  # ● P2 工具函数
│   └── types/                  # ● P2 类型定义
├── config/                     # ▲ P1 构建配置
├── public/                     # ● P2 静态资源
├── tests/                      # ● P2 测试文件
├── scripts/                    # ● P3 辅助脚本
└── package.json                # ▲ P0 依赖清单
```

## 代码文件内容分析

### 核心文件摘要（每个关键文件按此结构分析）

- **文件职责**：一句话说清这个文件解决什么问题
- **核心导出**：对外暴露的函数/类/组件/类型
- **内部依赖**：引用了哪些项目内的其他模块
- **被依赖方**：被哪些模块引用（评估影响范围）
- **关键逻辑**：文件中最需要理解的核心代码段
- **潜在问题**：明显的技术债、性能隐患、安全风险
- **修改建议**：如果重构应从何处入手

### 分析范围

- P0 文件（入口、路由、配置）：逐文件完整分析
- P1 文件（核心模块、服务层）：列出全部接口和职责
- P2 文件（组件、工具）：列出名称和用途摘要
- P3 文件（样式、测试、脚本）：归类说明，不逐文件分析

## 埋点与追踪

### 埋点 SDK 识别

- 搜索关键词：`gtag`, `mixpanel`, `sensors`, `sentry`, `aegis`, `analytics`, `track(`, `trackEvent`, `logEvent`, `pageView`, `reportEvent`
- 埋点 SDK 初始化位置
- 埋点配置（appId、reportUrl、采样率等）

### 埋点调用整理

- 搜索所有埋点/上报调用点
- 整理每个埋点的事件名、触发条件、所在文件、参数
- 按类别分组：页面浏览、用户行为、业务流程、错误事件
- 标注已验证和待确认的埋点
- 识别关键流程中缺失埋点的环节

### 埋点合规检查

- 是否包含敏感信息上报
- 是否有用户退出机制
- 是否区分了测试/生产环境

## 快速导航索引

- 按功能模块的跳转索引（如：用户认证 → LoginPage, AuthService, userStore）
- 按文件类型的速查表（入口文件、API 服务、组件、工具函数）
- 常用命令速查（dev, build, test, lint, deploy）
- 核心流程图路口（从哪个页面/API 开始追踪）

## 新人入门

- 第一个应读文件
- 第一个应运行命令
- 第一个应测试的 UI/API 流程
- 第一个安全的小改动
- 初期应避免修改的文件
- 修改核心逻辑前需要理解的概念
